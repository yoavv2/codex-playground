# Phase 01 Plan: Prep and Decisions

## objective

Lock the Phase 01 decisions needed to start implementation safely and efficiently:
- confirm repo strategy (Option A vs B)
- confirm remote transport path (Tailscale)
- validate Farfield repo availability/readiness for Phase 02
- define the MVP-blocking approval prompt API scope
- record decisions and blockers in planning artifacts for Phase 02 handoff

This plan is intentionally execution-focused and should complete without writing product code. Its main output is a validated implementation direction and updated planning context.

## execution_context

### Required reads (local planning)

- `.planning/PROJECT.md`
- `.planning/ROADMAP.md`
- `.planning/STATE.md`
- `.planning/config.json`

### Optional reads (only if present)

- `.planning/phases/01-prep-and-decisions/01-CONTEXT.md`
- `.planning/codebase/*` (if a codebase map exists)

### Discovery mode

- Base discovery level: `Level 1` (planning/docs + local environment)
- Upgrade to `Level 2` if a local Farfield repo/path is available (inspect repo structure and relevant API surfaces)
- Do not use network research in this phase unless the user explicitly requests it

### Tooling expectations

- Prefer `rg`, `ls`, `find`, and `git` for discovery
- Avoid destructive commands
- If a repo path is missing, ask once and continue documenting defaults/blockers if no answer is available

## context

### Phase metadata

- GSD phase: `01`
- Phase name: `Prep and Decisions`
- Milestone: `Milestone 1 - Secure Remote Foundation`
- Roadmap goal: lock repo strategy, transport path, and implementation defaults before code changes

### Confirmed project constraints (from state/project)

- The mobile app is a remote controller for Codex running on a Mac
- Mobile app communicates with Farfield over HTTP + SSE
- Security hardening must happen before real remote use
- Tailscale is the preferred remote access path (to confirm in this phase)
- Codex approval prompts (`command`, `file-change`, `apply-patch`) are MVP-blocking for practical remote usage

### Pending decisions to resolve in this phase

- Repo strategy: Option A (`Farfield` fork + `apps/mobile`) vs Option B (separate mobile repo)
- Exact Farfield fork/branch or local repo path to target for Phase 02
- Initial SSE library choice for React Native (or explicit defer-with-criteria)
- Approval prompt API shape (pending approvals + approve/deny actions) to unblock Phases 04-05
- Additional remote-mode hardening scope to include in Phase 02 (for example rate limiting, health auth)

### Phase 01 success target

By the end of this phase, Phase 02 should be executable without re-litigating major architecture/security decisions.

## tasks

### Task 1: Hydrate planning context and extract decision checklist (mandatory discovery)

1. Read and summarize the current Phase 01 requirements from:
   - `.planning/PROJECT.md` (Phase 0 / prep section, security requirements, repo strategy options)
   - `.planning/ROADMAP.md`
   - `.planning/STATE.md`
2. Build a concrete decision checklist for this phase with three buckets:
   - `lock now`
   - `defer intentionally`
   - `blocked on missing repo/path`
3. Confirm whether a prior phase context file exists:
   - if `.planning/phases/01-prep-and-decisions/01-CONTEXT.md` exists, merge its decisions/notes into the checklist
   - if not, plan to create it in Task 4

Deliverable for this task:
- A working checklist (kept in notes during execution, later written to `01-CONTEXT.md`)

### Task 2: Perform Farfield repo discovery and readiness check (Level 1 -> Level 2 if possible)

Goal: determine whether Phase 02 can start immediately and whether Option A is practical in the current environment.

1. Detect whether a Farfield repo/fork is already available locally.
   - Search likely nearby directories (non-destructively) and/or inspect user-provided paths if known.
2. If a local repo is found, inspect and record:
   - repository root path
   - current branch
   - monorepo/package layout (especially `apps/server`, `apps/web`, `packages/protocol`, and whether `apps/mobile` can fit cleanly)
   - location of server entrypoints and routes relevant to `/api/*`, `/events`, and `/api/debug/*`
   - evidence of approval prompt protocol/state support (or lack of it)
3. If no local repo is found:
   - mark Phase 02 as blocked on repo path/fork selection
   - continue the rest of this phase using planning-only outputs
   - record exactly what input is needed from the user (repo path, repo URL, preferred fork, branch)

Deliverable for this task:
- Repo readiness notes in `01-CONTEXT.md` (including explicit blocker if no repo is available)

### Task 3: Finalize Phase 01 decisions and defaults

Use the project brief + discovery results to resolve decisions with bias toward execution.

1. Repo strategy decision
   - Default to Option A (`Farfield` fork + `apps/mobile`) unless discovery shows it is impractical
   - If choosing Option B, record why and what coordination costs are accepted
2. Remote transport decision
   - Confirm Tailscale as the required remote path for MVP/personal use
   - Explicitly reject public internet exposure/port forwarding for MVP
3. Implementation defaults decision set
   - Lock Expo (TypeScript), Expo Router, TanStack Query, `expo-secure-store`
   - Decide whether SSE client choice is:
     - `locked` (specific package), or
     - `deferred` with decision criteria and validation task in Phase 03
4. Approval prompt API scope decision
   - Define minimum API capabilities needed for remote approval flows (list pending approvals, approve/deny action, thread association, prompt type)
   - Mark whether this belongs in Phase 02 (backend/protocol prep) and/or Phase 04 (client integration)
5. Remote hardening scope boundary for Phase 02
   - Confirm required: bearer token auth, `/events` auth, CORS allowlist, debug endpoint gating
   - Mark optional: rate limiting, health endpoint auth, sensitive log redaction

Deliverable for this task:
- Final decisions + explicit deferrals + blockers written to `01-CONTEXT.md`

### Task 4: Write/update planning artifacts for downstream phases

1. Create or update `.planning/phases/01-prep-and-decisions/01-CONTEXT.md` with:
   - phase goal (one paragraph)
   - confirmed decisions (final list)
   - deferred decisions (with owner phase and criteria)
   - blockers (if any)
   - repo readiness notes (path/branch/layout or missing-input request)
   - Phase 02 handoff notes (what to implement first)
2. Update `.planning/STATE.md`:
   - move resolved items out of `Pending Decisions (Phase 01)`
   - add newly discovered blockers/risks (if any)
   - refresh `Last Updated`
   - set `Next Action` to Phase 02 planning or to the blocker-resolution action
3. Update `.planning/ROADMAP.md` only if discovery changes the phase sequencing/scope materially.
   - Keep edits minimal and explain any change in the Phase 01 summary.

Deliverable for this task:
- Updated `01-CONTEXT.md`
- Updated `.planning/STATE.md`
- Optional minimal `.planning/ROADMAP.md` adjustments

### Task 5: Prepare execution handoff for Phase 02

1. Produce a concise handoff section in the Phase 01 output artifacts covering:
   - target repo/path (or blocker)
   - exact Phase 02 implementation scope
   - required environment variables to implement/document (`FARFIELD_AUTH_TOKEN`, `FARFIELD_ALLOWED_ORIGINS`, `FARFIELD_ENABLE_DEBUG_API`, optional health auth flag)
   - acceptance test outline for auth + `/events` + debug gating
2. Ensure Phase 02 planning can proceed without reopening Phase 01 decisions.

Deliverable for this task:
- Phase 02-ready handoff notes in `01-CONTEXT.md` and Phase 01 summary
