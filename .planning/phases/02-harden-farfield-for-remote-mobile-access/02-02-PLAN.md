- implement a pending approvals read path
- implement approve/deny action handling (or, if blocked by service/protocol gaps, add the missing server/service plumbing and document the residual gap)
- document and verify the resulting API contract for mobile use

## context

### Phase linkage

- Phase 02 hardening and auth/CORS assumptions come from `02-01`
- Approval prompt support is MVP-blocking per project brief and Phase 01 decisions
- Mobile UI integration is planned for Phase 04, but backend API support must exist first

### Current verified server capabilities

- `GET /api/threads/:id/live-state` exists and returns `ownerClientId`, `conversationState`, and `liveStateError`
- `POST /api/threads/:id/user-input` exists for `request_user_input` submission
- No explicit approval endpoints are currently exposed in Farfield HTTP routes
- Codex protocol vendor schemas include approval request/response types (`ExecCommandApproval*`, `FileChangeRequestApproval*`, `ApplyPatchApproval*`)
- `packages/codex-api/src/service.ts` currently exposes `submitUserInput`, but no explicit approval action methods

### Output contract needed by mobile MVP

Minimum mobile-usable approval prompt contract:
- pending approvals list scoped to thread
- stable approval id
- approval type (`command`, `file-change`, `apply-patch`)
- status (`pending` / resolved state)
- display payload (summary + details needed for review)
- approve action
- deny action
- deterministic success/error response shape

## tasks

1. Inspect the `conversationState` shape in `@farfield/protocol` (`packages/codex-protocol/src/thread.ts`) and the reducer path in `packages/codex-api/src/live-state.ts`.
2. Verify whether pending approval prompts are already present in reduced `conversationState` payloads:
   - inspect types first
   - if needed, inspect representative runtime payloads or traces (respect repo privacy rules; do not commit raw traces)
3. Inspect `packages/codex-api` and `codex-agent` adapter for existing approval action plumbing.
4. Document findings in `02-CONTEXT.md` with a comparison table:
   - data already available
   - data missing
   - action methods already available
   - service/protocol plumbing required

Deliverable:
- Evidence-backed capability inventory for approval reads/actions

### Decision Checkpoint: Choose approval exposure strategy

Choose and record one strategy before implementing endpoints:

1. **Extend `GET /api/threads/:id/live-state`**
   - add normalized `pendingApprovals` in the response payload
   - keep a single polling/read path for mobile
2. **Add dedicated endpoint(s)** (for example `/api/threads/:id/pending-approvals`)
   - isolate approval-specific response contracts
   - reduce coupling to full live-state payload evolution

Decision criteria:
- amount of data already available in `conversationState`
- code churn and risk in existing `live-state` consumers
- mobile polling simplicity
- schema clarity and strict typing

Record the chosen strategy in `02-CONTEXT.md` and continue with the matching implementation path.

### Task 2: Implement pending approvals read API

Implement the read path selected at the decision checkpoint.

1. Add/adjust server route handling in `apps/server/src/index.ts`.
2. Normalize approval prompt payloads into a strict mobile-facing shape:
   - stable id
   - type
   - thread id
   - status
   - summary/detail fields for UI rendering
3. If request params/body schemas are needed, add strict Zod schemas in `apps/server/src/http-schemas.ts`.
4. Ensure auth/CORS behavior from `02-01` applies to the new or modified route.

Deliverable:
- Mobile-usable pending approvals read path in Farfield HTTP API

### Task 3: Implement approve/deny actions (server + adapter/service plumbing as needed)

1. Determine the required action route shape (single endpoint with action enum vs separate approve/deny endpoints).
2. Add strict request body schema(s) in `apps/server/src/http-schemas.ts`.
3. Implement server route(s) in `apps/server/src/index.ts`.
4. Wire through `apps/server/src/agents/types.ts` and `apps/server/src/agents/adapters/codex-agent.ts`.
5. If `packages/codex-api/src/service.ts` lacks needed methods, add explicit approval action methods using existing protocol request/response schemas.
6. Preserve strict typing and Zod-validated boundaries per Farfield `AGENTS.md`.

Deliverable:
